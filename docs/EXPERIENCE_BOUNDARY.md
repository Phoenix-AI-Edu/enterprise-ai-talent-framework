# 平台模組邊界定義 (Experience Boundary)

為了確保 `AI_Talent` 平台在不斷擴充的過程中，維持清晰的商業漏斗與穩定的系統架構，我們嚴格定義以下四大模組的物理邊界與功能定位。未來的開發與上架作業，必須遵循此邊界規範，避免平台淪為難以維護的「大雜燴」。

---

## 1. 案例 (Cases)
**目錄路徑**：`AI_Talent/cases/` (包含 `cases.json` 與相關 HTML)
- **定位**：過去顧問輔導或專案執行的靜態成果展示。
- **目標受眾**：希望了解我們實力、建立信任感的企業客戶。
- **內容特性**：純靜態的文字、圖片、痛點描述與 ROI 數據。
- **邊界限制**：**不包含** 任何可互動的程式碼、後端 API 或動態表單。這層僅作為「閱讀型」的型錄。

## 2. 解決方案 (Solutions)
**模組關聯**：主站的 `方案 A/B/C` (對應首頁或獨立定價頁面)
- **定位**：顧問公司可銷售的標準化服務包或授權軟體。
- **目標受眾**：已經有明確需求，準備進入採購流程的客戶。
- **內容特性**：規格表、服務流程、價格區間、交付項目。
- **邊界限制**：這是在銷售端的定義，而非技術層面展示。它負責承接從 `cases` 或 `experience` 導流過來的商機。

## 3. 互動展示入口 (Experience Portal)
**目錄路徑**：`AI_Talent/experience/` (靜態入口頁面)
- **定位**：收納所有 AI 解決方案的「體驗大廳」。
- **目標受眾**：對特定 AI 技術感興趣，希望「親手玩玩看」的潛在客戶。
- **內容特性**：輕量級的前端頁面，負責展示 Solution Page (痛點、解法、Demo 影片)，以及一個導向外部互動系統的入口按鈕。
- **邊界硬規則 (Hard Rules)**：
  - **純靜態架構**：這裡只能有 HTML、JS、CSS 與 JSON 資料模型 (`solutions_data.js`)。
  - **拒絕後端與邏輯**：嚴禁放置 FastAPI、Node.js 後端程式碼。
  - **拒絕資料庫**：嚴禁放置 `.db` 或任何持久化資料檔案。
  - **拒絕敏感資訊**：嚴禁放置金鑰 (`.env`) 或任何真實客戶資料 (CSV/JSON)。
  - **拒絕原始碼**：絕不把外部 Demo App 的原始碼 (Source Code) 塞進主站 Repo 內，主站只提供外部導流連結。
  - **絕對隔離**：**嚴禁** 在此目錄下放置 FastAPI 後端程式、SQLite 資料庫 (`.db`) 或任何容器化 (`Docker`) 相關設定。

## 4. 互動展示系統 (Demo Apps)
**目錄路徑**：各自獨立的 GitHub Repositories (如 `central-kitchen-ai-agent`)
- **定位**：真實運作的 MVP 軟體系統，供客戶實機操作。
- **部署位置**：獨立的雲端環境 (如 Google Cloud Run, Render, Zeabur, 或專屬 VPS)。
- **外部存取**：擁有獨立的網址 (例如 `https://demo-central-kitchen.xxx`)。
- **邊界限制**：
  - **與主站解耦**：如果 Demo App 崩潰、過載或遭到攻擊，`AI_Talent` 主站必須毫髮無傷。
  - **具備自我清理能力**：必須實作排程重置 (Cron Reset) 機制，確保髒資料不會跨日留存。
  - **資料隔離**：不使用真實客戶的 PII (個人身分資訊)，全數採用去識別化的 Seed Data。

## 邊界總結矩陣

| 模組名稱 | 存放位置 | 內容型態 | 系統風險 | 核心目的 |
| :--- | :--- | :--- | :--- | :--- |
| **Cases** | 主站 Repo (`/cases`) | 靜態圖文 | 零 | 建立信任、火力展示 |
| **Solutions** | 主站 Repo (首頁) | 靜態圖文 | 零 | 引導採購、方案定價 |
| **Experience** | 主站 Repo (`/experience`) | 靜態入口 | 極低 | 解決方案型錄、名單收集 (Lead Gen) |
| **Demo Apps** | 獨立 Repo + 雲端部署 | 動態全端 App | 中高 (需防護) | 互動體驗、技術證明 (PoC) |
