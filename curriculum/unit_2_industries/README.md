## 📋 10 大常見 AI 基礎應用功能模組

|---|---|---|---|

| **4. 製造業預測性維護** | 機台故障預警、預防停機。 | 標記資料極度缺乏、感測器高頻雜訊。 | 特徵工程 + 異常檢測模型 |

## 🤖 2.11 Agentic RAG (90 分鐘)

### 1. 痛點 ➡️ 為什麼傳統 RAG 不夠用了？
傳統 RAG 採用「單次檢索（One-shot Retrieval）」模式，在面對簡單的知識問答（如：*「A型扣件的扭力是多少？」*）時表現優異。但在面對企業跨部門的複雜綜合查詢時，傳統 RAG 會瞬間癱瘓。

> [!NOTE]
> **📊 傳統 RAG 失敗案例 (孟顧問輔導實績匿名化)**
> 某半導體封測廠導入了傳統 RAG 知識庫。
> 研發工程師問：*「我們在 2024 年針對『打線封裝（Wire Bonding）』所收到的主要客訴原因是什麼？這跟我們 Q3 採購的 A 廠金線張力規格有關聯嗎？」*
> **傳統 RAG 的回答**：僅列出打線封裝的客訴定義，無法給出關聯性。
> **為什麼？** 因為該問題需要：
> 1. 解構問題，分別檢索「2024 客訴資料庫」、「金線張力規格書」、「Q3 採購明細」。
> 2. 將金線張力數據與客訴主因進行時間序列的**交叉比對**。
> 3. 計算出張力異常與客訴率的相關係數。
>
> 傳統 RAG **只能進行單一文檔的模糊檢索，無法進行規劃與邏輯計算**。

### 2. 技術 ➡️ Agentic RAG 核心架構
Agentic RAG 將 RAG 從「模型」升級為「主動代理人系統 (Multiagent System)」，核心架構如下：

```
                ┌──────────────────────────────────────┐
                │          工程師複雜查詢              │
                └──────────────────┬───────────────────┘
                                   │
                                   ▼
                ┌──────────────────────────────────────┐
                │      規劃代理 (Planning Agent)       │
                │      將複雜問題分解為多個子任務      │
                └──────────────────┬───────────────────┘
                                   │
                                   ▼
            ┌──────────────────────┼──────────────────────┐
            ▼                      ▼                      ▼
┌──────────────────────┐┌──────────────────────┐┌──────────────────────┐
│  檢索代理 (RAG Tool) ││ 結構化 SQL 查詢代理  ││   計算代理 (Python)  │
│ 撈取客訴紀錄PDF      ││ 撈取金線張力規格數據 ││ 計算數據相關性與時序 │
└──────────┬───────────┘└──────────┬───────────┘└──────────┬───────────┘
           │                      │                      │
           └──────────────────────┼──────────────────────┘
                                   ▼
                ┌──────────────────────────────────────┐
                │      整合代理 (Synthesis Agent)      │
                │      交叉比對各管道數據並初步總結    │
                └──────────────────┬───────────────────┘
                                   │
                                   ▼
                ┌──────────────────────────────────────┐
                │       審查代理 (Review Agent)        │
                │     自我批判 (Self-Critique) 與驗證  │
                └──────────────────┬───────────────────┘
                                   │
                                   ▼
                ┌──────────────────────────────────────┐
                │     最終回答（精確引用文檔與數據）   │
                └──────────────────────────────────────┘
```

#### 4 大關鍵技術元件：
* **Query Decomposition (問題分解)**：AI 自動將複雜問題拆解為多個子問題，分流給不同的代理人執行。
* **Multi-hop Retrieval (多跳檢索)**：第一輪檢索的輸出，自動作為第二輪檢索的關鍵字輸入，在多份無關聯的文檔間跳躍尋找線索。
* **Self-Critique (自我批判)**：審查代理人檢查回答的準確度，若發現「無法完全回答使用者問題」或「引用來源有出入」，會自動退件並重啟檢索。
* **Tool Use (工具調用)**：AI 可主動調用 Python 直譯器、SQL 資料庫 API 進行計算，而非單純靠 LLM 的文字預測。

### 3. 應用 ➡️ 企業 5 大導入場景
1. **跨部門研發與專利文獻挖掘**：跨越研發、專利、生產線的深度交叉查詢。
2. **智慧合約風險審查**：一鍵比對新合約與歷史訴訟案例的潛在違約風險。
3. **複雜客服分流與工單處理**：整合客戶歷史訂單、庫存 API 與售後服務政策進行自動答覆。
4. **國際合規查檢自動化**：對齊 ISO 42001 與 NIST 框架，自動查檢內部管理手冊的缺口。
5. **財務多源報表交叉審計**：自動比對 ERP 數據、出貨單 PDF 與發票明細。

### 4. 採購 ➡️ 工具堆疊與決策樹
#### 2026 主流 Agentic RAG 工具堆疊對比：
* **LangGraph** (Microsoft/LangChain)：最適合中大型企業，支援極其複雜的「有向有環圖 (DAG)」狀態管理。建置成本高 (NT$ 80-200萬)，高學習曲線。
* **CrewAI**：學習曲線中等，適合快速建置跨部門的角色代理人協作。
* **n8n + AI Nodes (低代碼)**：適合新手企業或無 ML 團隊之中小企業，透過視覺化工作流低成本起步。

```
【RAG 升級決策樹】
目前 RAG 使用率 > 30% 且面臨跨文檔查詢痛點？
  ├─ 是 ➡️ 企業是否有自建開發團隊？
  │         ├─ 是 ➡️ 優先採用 LangGraph 或 CrewAI 進行深度自研。
  │         └─ 否 ➡️ 優先採用 n8n 低代碼方案，或採購鳳凰 AI 陪跑服務。
  └─ 否 ➡️ 暫不建議升級，應先優化現有傳統 RAG 的資料清洗與 Chunking。
```

### 5. 風險 ➡️ 5 大落地紅線
* **TCO 成本失控**：Agentic RAG 的多輪檢索會導致每次查詢呼叫 5-10 次 LLM，Token 耗費暴增。必須在程式中強制加入 **「最大循環限制 (Max Iterations / Loop Guard)」**，防止 Agent 因為 Tool Call 失敗而陷入死循環，一夜之間燒光企業的 API 額度。
* **回應延遲太長**：由於包含多輪迭代與自我批判，回答時間通常需 10-30 秒，**絕對不適合需要即時回應的線上聊天客服**。
* **Debug 難度倍增**：決策路徑太長，難以定位是檢索出錯、規劃出錯還是合成出錯。
* **數據品質依賴**：如果企業的原始 PDF 和資料庫是一堆垃圾（Garbage in），Agent 再聰明也只能產出垃圾（Garbage out）。
* **過度工程化**：簡單的 FAQ 場景使用 Agentic RAG 是「大砲打小鳥」，白白浪費 API 費用。

### 6. 案例 ➡️ 半導體封測廠跨部門研發與專利挖掘
* **真實案例 (PHX-CASE-2026-004)**：南部某半導體封測大廠，過往研發工程師在開發新封裝製程時，需要手動去翻閱 20 年累積的 5,000 份測試報告、1,200 份客戶客訴單、以及外部 10 萬件專利文獻。
* **導入方案**：鳳凰 AI 協助其基於 **LangGraph + LlamaIndex** 建立了「Agentic RAG 製程研發決策助理」。
* **實質 ROI**：
  * 研發前期文獻與歷史瑕疵檢索時間**從 5 天縮短至 15 分鐘**。
  * 新產品開發決策週期**縮短 30%**。
  * **成功避免了一次專利侵權風險**（AI 主動在第三跳檢索中比對出某競爭對手於歐盟的專利案）。
  * 專案建置成本：NT$ 180 萬，折抵政府 SBIR 補助後，企業實質淨支出僅 NT$ 90 萬（對接 [單元八補助計畫](file:///g:/我的雲端硬碟/AI_Talent/curriculum/unit_8_grants/README.md)）。

## 🎙️ 2.12 Voice AI 與 Realtime API (75 分鐘)

### 1. 痛點 ➡️ 為什麼語音是傳產 AI 落地的「唯一破口」？
許多傳統製造業在推動數位轉型或 AI 導入時，常強推「現場師傅打字輸入工單或查詢系統」，結果使用率幾乎為零。

> [!NOTE]
> **💥 傳產調機現場的真實死穴**
> 「師傅的雙手沾滿了黃油、黑油，拿著厚重的扳手與量規。你讓他在吵雜的廠區，脫下防護手套，在髒污的平板電腦或手機螢幕上打字輸入瑕疵代碼？這是不切實際的數字幻覺。」
>
> 孟顧問的輔導實績表明：**傳統製造業一線員工對打字系統的抗拒度高達 90% 以上。但如果改成「口頭說話」，AI 自動辨識並結構化建檔，使用率會瞬間暴增！**

### 2. 技術 ➡️ 2026 語音技術革命：Realtime API
過往的語音 AI 採用 **「串接式架構」**：`語音 ➡️ STT ➡️ LLM ➡️ TTS ➡️ 播放`。這會產生 3 至 8 秒的致命延遲，且無法中斷。
2025-2026 年，以 **OpenAI Realtime API** 與 **Gemini Live API** 為代表的「端到端語音模型」問世，徹底改寫了遊戲規則。

#### 傳統串接 vs. Realtime API：
* **傳統串接 (STT+LLM+TTS)**：延遲 3-8 秒。音色生硬，像機器人說話。無法打斷。每分鐘成本約 NT$ 1.5 - 3。
* **Realtime API (端到端音訊)**：延遲 **0.5-1.2 秒（與真人對話無異）**。音色充滿情緒起伏、呼吸聲與語調抑揚頓挫。**支援隨時打斷（Interruption）**。每分鐘成本約 NT$ 5 - 15。

### 3. 應用 ➡️ 廠區重工與服務業 4 大用例
* **語音即時交班與異常回報**：師傅口述 30 秒，AI 自動翻譯、結構化為標準異常報告並寫入 ERP。
* **免動手語音 SOP 查詢**：技師雙手正在維修機台，直接開口提問，AI 即時語音朗讀該機台的維修手冊步驟。
* **智慧語音電話客服 (AI Agent Call Center)**：低延遲、具情緒感知能力地處理大量重複性客戶預約與查詢。
* **多語言現場即時接待**：零售業、餐飲業的 AI 語音翻譯櫃檯。

### 4. 採購 ➡️ 台灣現場的三大在地化控制點
企業在採購 Voice AI 系統時，必須注意以下台灣特有的落地細節：
* **方言與口音支援**：系統必須具備強大的「國台語混雜」辨識能力，特別是台灣南部特有的產業術語口音（如：*「這台 controller 目前有 lens 誤差，愛去 check 一下」*）。
* **降噪與回音消除 (Noise Cancellation)**：製造業廠區的沖床、鍛造機噪音高達 85dB 以上。必須搭配「陣列式降噪麥克風」與「邊緣端語音活動檢測 (VAD)」，否則 AI 將聽不清語音。
* **個資保護與身分揭露**：依照台灣個資法，語音錄音屬於敏感個資。系統必須內建自動去識別化，且在對話前**強制主動用語音告知「您好，我是 AI 語音助理，為了優化服務，本次對話將進行加密記錄...」**。

### 5. 風險 ➡️ 3 大防線
* **Realtime API 成本高昂**：若 24 小時開啟，流量費用會非常驚人。必須設計 **「休眠與喚醒機制（如語音關鍵字喚醒）」**。
* **噪音環境的誤觸發**：廠區的高頻鐵器碰撞聲容易被誤判為人類語音，導致系統不斷觸發。
* **複雜業務無法閉環**：當客戶在電話中提出高度複雜的客訴時，AI 若強行回答會造成公關災難。必須建立 **「無縫轉接真人 (HITL)」** 機制。

### 6. 案例 ➡️ 精密扣件廠語音交班 & 重工業品質異常回報

#### 案例 A：精密扣件廠現場語音交班 (PHX-CASE-2026-001)
* **背景**：南部某精密扣件廠，現場調機師傅平均年齡 52 歲，不願使用手機打字交班，導致交班日誌經常缺漏，設備瑕疵數據難以追蹤。
* **方案**：導入基於 **Gemini Live API** 的廠區語音對講機系統。師傅交班時只需按住對讲機說：*「今天第三台螺絲打頭機送料夾有點偏，我已經先微調了，下班要注意看有沒有偏芯。」*
* **ROI**：
  * 師傅交班日誌填寫率 **從 12% 暴增至 98%**。
  * 瑕疵數據收集週期從 1 週縮短至 **當日即時**。
  * 成功提前預警了 3 次設備停機事件，**避免損失 NT$ 45 萬**。

#### 案例 B：精密模具與鋼鐵重工廠區異常通報 (PHX-CASE-2026-002)
* **背景**：高雄某大型金屬鑄造與模具工廠，高爐與行車現場環境極其吵雜且高溫。
* **方案**：配置內建主動降噪演算法的頭盔式麥克風，搭配本地端輕量級語音識別模型 (Whisper Local + SLM)。
* **ROI**：
  * 在 90dB 的高噪音環境下，**語音異常通報精準度達 93%**。
  * 品質異常通報流程從「師傅手寫 ➡️ 助理打字 ➡️ 主管簽核（歷時 4 小時）」縮短為 **「口頭回報 ➡️ AI 自動結構化建檔 ➡️ 主管手機即時簽核（歷時 2 分鐘）」**。

## 🖥️ 2.13 Computer Use Agent (60 分鐘)

### 1. 痛點 ➡️ 跨系統資料搬運的行政黑洞
許多企業的行政、財務與 HR 員工，每天要花費 20-40% 的時間在不同的系統（如 SAP 系統、Excel、LINE、企業信箱）之間進行重覆性的「資料複製與貼上」工作。傳統的 RPA (機器人流程自動化) 雖然能解決部分問題，但建置成本極高、且只要系統 UI 有微小變動（如按鈕往右移了 2 像素），RPA 腳本就會失效。

### 2. 技術 ➡️ 2026 劃時代技術：Computer Use
**Computer Use** 允許 AI 像人類一樣，直接「觀看」電腦螢幕（截圖解析）、移動滑鼠、點擊按鈕、並用鍵盤輸入文字。

```
  [ 使用者自然語言指令 ] ➡️ AI 觀看螢幕截圖 ➡️ 規劃下一步 ➡️ 移動滑鼠/點擊 ➡️ 再次截圖確認
```

例如，Anthropic 的 **Claude Computer Use API** 或是 OpenAI 的 **Operator**，允許開發人員給予一個簡單指令：*「幫我把 Excel 裡的新增客戶名單，手動登錄到我們公司的 SAP 舊系統中，並在完成後寄信通知主管。」* AI 會自行打開 Excel、讀取內容、打開網頁、點擊登錄按鈕、並打開 Outlook 完成發信。

### 3. 應用 ➡️ 試點階段的三大安全情境
由於該技術目前尚不夠成熟，鳳凰 AI 僅建議企業在以下 **「低風險、高度人工覆核 (Human-in-the-Loop)」** 的場景進行試點：
* **跨系統靜態資料搬運**：無邏輯判斷的單純登錄工作。
* **週報/月報自動化生成**：AI 自動登入多個系統撈取數據，填入 PowerPoint 或 Word 範本。
* **測試自動化**：軟體系統上線前，由 AI 模擬使用者點擊進行 UI 測試。

### 4. 採購 ➡️ 為什麼鳳凰 AI 警告「現階段不建議大規模採購」？
作為一家具備極高商業倫理與專業度的顧問品牌，**鳳凰 AI 必須對企業客戶誠實揭露 Computer Use 目前致命的 5 大落地風險**：

> [!WARNING]
> **🚨 Computer Use 目前的五大致命風險**
> 1. **成功率依然不足**：2026 年最新實測表明，簡單的單系統操作成功率約 85-90%；但一旦涉及「跨 3 個以上系統、且有邏輯分支判斷」的複雜任務，**成功率會暴跌至 30-50%**。這代表每 2 次操作就有 1 次會出錯，可能造成財務或資料庫的災難。
> 2. **執行速度極慢**：人類員工點擊一個按鈕只需 0.5 秒；但 Computer Use Agent 每次動作都需要進行「截圖 ➡️ 高維向量分析 ➡️ LLM 推理 ➡️ 傳送滑鼠 API」，**完成一個動作需要 5 至 15 秒**。在大量搬運資料時，速度甚至比人類手動操作還要慢。
> 3. **極高的資訊安全漏洞 (Prompt Injection)**：如果 AI 在操作電腦時，打開了某個含有惡意代碼的客戶網頁，惡意網頁可以透過 **「提示詞注入 (Prompt Injection) 攻擊」** 奪取 AI 的控制權，指示 AI 刪除硬碟資料、或將企業機密資料透過 Email 偷偷寄給駭客。
> 4. **責任歸屬與 ToS 違規**：AI 代替員工操作大廠系統（如 SAP、Salesforce），一旦因為出錯造成資料損毀，責任由誰承擔？此外，許多 SaaS 廠商的服務條款 (ToS) 明文禁止「非人類自動化爬蟲操作」，企業可能面臨被停權封號的法律風險。
> 5. **TCO 財務失控**：每次滑鼠點擊都需要傳送一張高解析度的螢幕截圖給 LLM，Token 耗費極大。一個原本 NT$ 20,000 薪資的助理工時，若用 AI 代替，**單月的 API Token 帳單可能高達 NT$ 50,000**，產生嚴重的「財務逆選擇」。

### 5. 鳳凰 AI 獨家：採購決策防護欄

## 🎬 2.14 多模態影片生成 (75 分鐘)

### 1. 痛點 ➡️ 企業數位內容產出的成本懸崖
在影音社群時代，企業在行銷廣告、員工 SOP 培訓、品牌公關上面臨著海量的影片內容需求。然而，傳統影片拍攝 average 成本高達 **每分鐘 NT$ 30,000 至 100,000**，且只要產品更新，整支影片就必須重新拍攝，這對企業形成了沉重的財務負擔。

### 2. 技術 ➡️ 2026 影片生成 AI 全景
2026 年，影片生成 AI 已經跨過了畫質模糊的階段，達到了「影視級與寫實主播」的商用成熟度。

#### 兩大商用主流方向：
* **寫實創意生成 (Sora 2, Runway Gen-4, Google Veo 3)**：
  * **特點**：從文字 (Text-to-Video) 或圖像 (Image-to-Video) 直接生成高解析度、物理規律準確的 3D 影片。
  * **商用價值**：替代高成本的 3D 動畫、空拍畫面、以及概念行銷短片。
* **AI 虛擬主播平台 (HeyGen, Synthesia)**：
  * **特點**：只需輸入文字腳本，即可由極度寫實的「AI 語音替身主播」進行多國語言的簡報、教學與新聞朗讀。
  * **商用價值**：新人培訓、SOP 講解影片產出的自動化。

### 3. 應用 ➡️ 培訓部與行銷部的 2 大實戰路徑
* **路徑 A：訓練部標準化培訓片（最有 ROI 價值）**：
  * 企業購買 HeyGen / Synthesia 服務，將既有的文字版產品規格書與 SOP 手冊，一鍵重製為「AI 寫實主播示範影片」。
  * **多語言一鍵生成**：一秒生成繁中、英文、越南文、印尼文版本，完美對接外籍移工的職前訓練。
* **路徑 B：行銷部高頻社群內容工廠**：
  * 利用 Sora 2 或 Runway 快速生成大量抖音 (TikTok)、Reels 或 YouTube Shorts 廣告短片素材。

### 4. ⚖️ 法律與倫理風險 (企業採購的黃金防線)
影片生成 AI 雖然好用，但企業一旦踩到法律紅線，會面臨巨額的公關災難與賠償：
* **著作權歸屬爭議**：目前多國法律（包含台灣）認定 **「單純 AI 生成、無人類實質創意介入之作品，不受著作權法保護」**。企業辛苦用 AI 生成的行銷影片，競爭對手可以直接複製使用而不違法。此外，商用前**務必檢視工具的 ToS (服務條款) 是否包含 IP 賠償承諾（如 Adobe Firefly 與 Runway 專業版有提供商用版權保護）**。
* **肖像權與 Deepfake 侵權**：
  * ❌ **絕對不能使用**：公眾人物或未授權真人的肖像與聲音進行生成。
  * ⚠️ **企業內部員工肖像授權**：若要將企業高階主管或優良員工作為 AI 語音替身 (Avatar) 使用，**必須強制簽署書面的《AI 肖像與語音替身永久商用授權同意書》**，防止員工離職後控告公司侵犯肖像權與姓名權。
* **虛偽不實廣告紅線**：
  依照台灣《公平交易法》第 21 條，商品之廣告不得有虛偽不實或引人錯誤之表示。
  * 🔴 **絕不能用 AI 生成的場景**：**「真人消費者見證影片」**（例如：用 AI 寫實主播偽裝成真實客戶說*「我吃了這家保健食品治好了我的...」*），這會被處以最高 **NT$ 2,500 萬的行政罰鍰**。
  * 🟢 **安全可用 AI 的場景**：產品外觀概念示意、創意背景、無見證屬性的單純產品功能講解。
* **AI 內容強制標示義務**：對齊 **Unit 8 第 3 章（歐盟 AI Act 標示義務）**，在 AI 生成的行銷影片角落，應主動標註 **「本影片包含 AI 生成內容」** 或採用 C2PA / Google SynthID 數位浮水印，建立負責任的企業形象。
