### Slide 03｜劃時代技術：2026 Model Context Protocol (MCP) 上下文協定革命
* **【版面設計】**：Apple 經典金色與深藍色搭配。中央為一個 USB 3.0 插頭的金色圖示，左側為「各種外部 LLM」，右側為「企業核心資產（SQL、Slack、GitHub、ERP）」，插頭居中標示 "Model Context Protocol"。
* **【核心標題】**：**大模型跨系統連接的插拔革命**
* **【核心內容】**：
  * **傳統對接痛點** ➡️ 每次要讓大模型調用新工具，工程師就必須為不同的 API 重寫客製化 glue code，開發成本極高。
  * **MCP 核心概念** ➡️ 由 Anthropic 等巨頭推動的 Model Context Protocol 協定，為 AI 建立了像 USB 般的標準化界面。
  * **三大支柱協定**：
    *   `Resources` ➡️ 標準化靜態數據讀取。
    *   `Tools` ➡️ 標準化動態代碼與外部函數執行。
    *   `Prompts` ➡️ 標準化預設提示詞模板。
* **【主講人】**：陳文家策略長
* **【時間管控】**：180 秒 (累計 07 分鐘)
* **【講師逐字稿】**：
  「大家好，我是陳文家。剛才孟顧問談到了 Agent 的失控，我將從技術底層為各位解析 2026 最具革命性的關鍵技術：**Model Context Protocol (MCP)**。

  在過去，如果您的 IT 團隊要讓 ChatGPT 讀取公司的 Oracle 資料庫、再把結果發送到 Slack，工程師必須針對這兩個軟體，用 Python 寫上千行的客製化代碼。當大模型一升級，這些代碼可能就壞了，IT 部門天天都在修補 Bug。

  MCP 協定徹底終結了這個噩夢。它就像電腦的 USB 接口。

  不論您用的是 Claude、Gemini 還是私有化 Llama-4，只要它們支援 MCP 標準，您就可以像插隨身碟一樣，一秒將大模型連接到 PostgreSQL、GitHub 或是既有的 ERP 中。這意味著：**AI 對接跨系統工具的開發成本，直接暴降了 80%！**」

### Slide 04｜MCP 對接架構：像 USB 一樣熱插拔降低 80% 開發成本
* **【版面設計】**：清晰簡潔的架構流程圖。左側為「MCP Client (例如大模型中台)」，中間為標準化的「MCP JSON-RPC 通信通道」，右側為「MCP Server 實體（連接 SQL、Notion、本地 AST 代碼庫）」。
* **【核心標題】**：**MCP 協定地端網路拓撲架構**
* **【核心內容】**：
  * **非對稱安全隔離**：MCP Client 與 MCP Server 運行於地端安全網閘（DMZ）內。
  * **工具宣告與調用 (Tool Calling)**：
    *   *Step 1* ➡️ MCP Server 向 Client 宣告可用工具清單與參數 JSON Schema。
    *   *Step 2* ➡️ 大模型判定意圖，回傳包含 `call_tool` 指令之 JSON 封包。
    *   *Step 3* ➡️ 本地 Server 物理執行代碼，回傳 `result` 文本，LLM 自身無直接執行權限。
* **【主講人】**：陳文家策略長
* **【時間管控】**：120 秒 (累計 09 分鐘)
* **【講師逐字稿】**：
  「這是我們寫入 [單元六完整教材 (curriculum_v2026.md) 中的標準 MCP 拓撲圖](file:///g:/%E6%88%91%E7%9A%84%E9%9B%B2%E7%AB%AF%E7%A1%AC%E7%A2%9F/AI_Talent/curriculum/unit_6_generative_ai/curriculum_v2026.md)。

  很多非技術背景的主管會擔心：大模型是不是直接接管了我們的伺服器，能隨便亂動我們的資料？

  答案是：**完全不行。這就是 MCP 的天才之處。**

  大模型（MCP Client）與我們的資料庫（MCP Server）之間，只有一條唯讀的 JSON 傳輸通道。大模型本身根本沒有直接執行 SQL 命令或刪除檔案的物理權限。

  當大模型判定需要查詢庫存時，它只能傳送一個『我申請使用查詢工具，參數是料號 A』的請求。本地的 MCP Server 收到請求後，在本地安全環境下執行查詢，再把文字結果告訴模型。

  大模型只是一個負責思考的『大腦』，它無法越過我們的 MCP 接口去擅自行動。這就在物理層面，建立了一道無法被突破的防火牆。」

### Slide 06｜第一道安全關卡：VM / Docker 沙盒物理隔離與權限最小化
* **【版面設計】**：深色背景。中央以發光效果呈現一個鋼鐵防禦堡壘的 VM/Docker 沙盒，內裝有 `Agent Executor`。外部以一把鎖標示「物理隔離與唯讀掛載」。
* **【核心標題】**：**安全防線一 ｜ 物理沙盒隔離架構**
* **【核心內容】**：
  * **代碼執行危險性** ➡️ Agent 在執行 Python 代碼或進行 AST (抽象語法樹) 代碼索引時，一旦失控會破壞伺服器作業系統。
  * **Docker Sandbox (沙盒物理隔離)**：
    *   **唯讀掛載** ➡️ Agent 僅能讀取指定的唯讀 VM 目錄，嚴禁訪問宿主機根檔案系統。
    *   **無網環境** ➡️ 執行計算的 Python Executor 沙盒完全物理切斷外網連線，防範惡意代碼外洩數據。
* **【主講人】**：陳文家策略長
* **【時間管控】**：120 秒 (累計 14 分鐘)
* **【講師逐字稿】**：
  「現在，我們進入最硬核的**『企業級 Agentic 五大安全防線』**。

  當我們允許 Agent 調用 Python 計算工具時，它會在伺服器後台自動生成並執行 Python 代碼。如果沒有沙盒隔離，一個寫錯的代碼、或者是惡意注入的指令：`import os; os.system('rm -rf /')`，會在一秒鐘內物理抹除您整台伺服器的檔案。

  **鳳凰 AI 的第一道關卡： Docker Sandbox 物理隔離。**

  我們把所有的代碼執行器（Agent Executor），全部關在一個物理上物理隔離、無外網連線的 Docker 容器中。

  這個容器對您的實體主機，只能『唯讀掛載』特定的靜態目錄。就算 Agent 被越獄攻擊、在容器內執行了惡意刪除命令，它能破壞的也只是一個拋棄式的虛擬沙盒，在 Session 結束的毫秒內就會被物理銷毀，宿主機自始至終毫髮無損。這就叫物理隔離。」

### Slide 07｜第二道安全關卡：Agentic Loop Guard 熔斷機制與 Max Iterations
* **【版面設計】**：發光的警報指示燈。展示 Agent 因 Tool Call 失敗而陷入死循環的「Token 暴燒路徑」，下方是一個金色的熔斷安全閘門 "Loop Guard Active"。
* **【核心標題】**：**安全防線二 ｜ TCO 財務熔斷控制**
* **【核心內容】**：
  * **致命財務危機** ➡️ Agentic 工作流在遭遇 Tool Call 網路抖動時，會非常負責任地判定「工具調用失敗，我必須變更參數重新重試！」
  * **Loop Guard 熔斷架構**：
    *   **Max Iterations = 5** ➡️ 強制限制單次任務重試不得超過 5 次。
    *   **Max Cost = US$ 2.00** ➡️ 限制單筆對話消耗 API 成本上限。
    *   **警報通道** ➡️ 超標時強制終斷 Session，物理隔離大模型連線，並向 Telegram/Slack 發送 IT 運維警報。
* **【主講人】**：陳文家策略長
* **【時間管控】**：180 秒 (累計 17 分鐘)
* **【講師逐字稿】**：
  「這是我們專門為 CFO 設計的**第二道安全防線：Loop Guard 熔斷機制**。這也是我們輔導時，CFO 最感激的一項設計。

  在普通的 RAG 或 Chatbot 中，使用者問一句，大模型答一句，Token 消耗是線性的。但 Agent 不一樣。Agent 是『自主運行』的。

  當它在幫您跑一段財務報表分析時，如果資料庫的某個欄位格式不對，大模型會非常負責任地自我修正：*『啊，SQL 查詢出錯了，我換個查詢語句重新試一次！』*

  如果沒有防護，它會在 1 秒鐘內嘗試幾十次，在幾分鐘內呼叫上萬次外部 API，產生無限循環。等您第二天上班，會看到一張驚人的數萬美元 API 帳單。

  我們在程式底層配置了 `Max Iterations` 與 `Max Cost` 限制。一旦單次對話重試超過 5 次、或成本超過 2 美元，系統會強制關閉 API 連線，直接熔斷。這就叫代碼級的預算控制。」

### Slide 08｜第三道安全關卡：既有 ERP/CRM 系統的「非侵入式唯讀副本」串接
* **【版面設計】**：流暢的系統拓撲圖。左側為「生產環境 SQL 核心庫（紅色警戒）」，中間為「一秒同步的 Replica 唯讀副本（發金色光芒）」，右側為「AI Agent」。
* **【核心標題】**：**安全防線三 ｜ 非侵入式唯讀數據防護**
* **【核心內容】**：
  * **核心痛點** ➡️ 企業核心 ERP/CRM 系統架構老舊（如 20年前的 SAP 舊系統），任何直接侵入式對接或直接 SQL 寫入都極易引發資料庫崩潰。
  * **零侵入式唯讀副本 (Read Replica)**：
    *   AI 所有關聯查詢、RAG 向量比對，只在**「物理隔離的 Read Replica 副本庫」**中執行。
    *   大模型自始至終接觸不到、也無物理權限直接對生產核心庫進行任何 `INSERT` 或 `DELETE` 操作。
* **【主講人】**：陳文家策略長
* **【時間管控】**：120 秒 (累計 19 分鐘)
* **【講師逐字稿】**：
  「很多 CIO 會跟老闆說：*『老闆，我們公司的 SAP ERP 已經用了 15 年，任何外部程式的直接寫入都可能導致系統死鎖（Deadlock），專案風險太大，我反對導入 Agent。』*

  這是非常合情合理的技術擔憂。
