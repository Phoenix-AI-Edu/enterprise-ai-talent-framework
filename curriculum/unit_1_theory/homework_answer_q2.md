# 📚 單元一課後練習解答報告：行銷自動化中的工具調用與 Agent 安全防禦機制

## 📋 評估題目背景
*   **場景**：公司的行銷部門希望 AI Agent 能自動根據 CRM 系統中的客戶歷史購買紀錄，發送客製化的促銷 Email。
*   **核心問題**：
    1. 結合「推理與工具調用（Tool Calling）」機制，定義給大模型使用的工具說明（API 函數名稱與參數）。
    2. 為了避免生成式 AI「自動發信」時產生幻覺（如捏造優惠券）或冒犯客戶，說明應在何處加入「鑑別式 AI 的意圖/內容審查」與「Human-in-the-loop (人工覆核)」安全機制。

---

## 🛠️ 一、 工具調用（Tool Calling）之 API 函數與參數定義

大語言模型（LLM）本身沒有數據庫存取權，也沒有發信能力。因此，我們必須為模型配備兩把「工具鑰匙」：
1. **讀取工具**：用於獲取 CRM 中該客戶的歷史消費數據。
2. **執行工具**：用於將生成好的促銷信件發送出去（此處會被安全機制攔截，詳見第二部分）。

以下是提供給 LLM 閱讀的工具說明定義（採用 JSON Schema 與中文虛擬碼描述）：

### 1. 工具一：取得客戶歷史購買紀錄 (Read CRM Data)
*   **函數名稱 (Name)**：`get_customer_purchase_history`
*   **功能描述 (Description)**：當行銷人員指定客戶 ID 時，此工具可從 CRM 資料庫讀取該客戶過去 12 個月內的所有交易紀錄、購買品項、消費金額與偏好標籤。
*   **參數說明 (Parameters)**：
    *   `customer_id` (字串, 必填)：客戶在系統中的唯一編號（例如：`USR-2026-889`）。

```json
{
  "name": "get_customer_purchase_history",
  "description": "取得特定客戶過去12個月的 CRM 歷史購買紀錄，包含商品名稱、類別、金額與購買時間，用以分析客戶偏好。",
  "parameters": {
    "type": "object",
    "properties": {
      "customer_id": {
        "type": "string",
        "description": "客戶的唯一識別識別碼，格式如 USR-XXXX-YYY"
      }
    },
    "required": ["customer_id"]
  }
}
```

### 2. 工具二：發送客製化促銷郵件 (Write/Execute Email)
*   **函數名稱 (Name)**：`send_marketing_email`
*   **功能描述 (Description)**：將指定內容與主旨的促銷郵件，發送給 CRM 系統中登記的該客戶電子信箱。
*   **參數說明 (Parameters)**：
    *   `customer_id` (字串, 必填)：收件客戶的 ID。
    *   `email_subject` (字串, 必填)：經 LLM 生成的吸引人郵件主旨。
    *   `email_body` (字串, 必填)：促銷郵件的 HTML/Text 正文內容，必須包含客製化推薦理由。
    *   `discount_code` (字串, 選填)：此次行銷活動要附帶的官方優惠代碼。

```json
{
  "name": "send_marketing_email",
  "description": "將生成的促銷郵件發送至該客戶的註冊信箱。注意：此工具在執行前會自動進入人工覆核佇列，不會立刻直接對外發送信件。",
  "parameters": {
    "type": "object",
    "properties": {
      "customer_id": {
        "type": "string",
        "description": "收件客戶的唯一識別碼"
      },
      "email_subject": {
        "type": "string",
        "description": "郵件標題，須具備吸引力且不包含誇大不實字眼"
      },
      "email_body": {
        "type": "string",
        "description": "郵件的客製化 HTML 正文，內容需提及客戶歷史購買的相關品項並推薦互補商品"
      },
      "discount_code": {
        "type": "string",
        "description": "行銷代碼（若有），如 PHOENIX2026"
      }
    },
    "required": ["customer_id", "email_subject", "email_body"]
  }
}
```

## 🛡️ 二、 確保安全的 AI Agent 架構設計 (安全防禦線)

為了防範 AI 產生**幻覺**（如捏造不存在的折扣、承諾錯誤的保單條款）或**品牌公關危機**（如使用冒犯性詞彙、過度打擾客戶），我們不能讓 LLM 擁有「直接發信」的最終執行權。

我們必須在系統架構中部署 **「雙層過濾閘門」**：

```text
  [ LLM 意圖推理 ] ➔ 呼叫 send_marketing_email
                          │
                          ▼
  ┌────────────────────────────────────────────────┐
  │  第一關：鑑別式 AI 內容審查 (Automatic Guardrails) │
  │  - 檢查是否包含敏感詞、冒犯詞                      │
  │  - 檢查折扣碼與金額是否在 CRM 政策白名單中         │
  │  - 比對幻覺（比對生成內容與 CRM 歷史購買的真實性）     │
  └──────────────────────┬─────────────────────────┘
                         │ Pass
                         ▼
  ┌────────────────────────────────────────────────┐
  │  第二關：人工覆核佇列 (Human-in-the-Loop, HITL) │
  │  - 於行銷後台系統生成「草稿待審」佇列               │
  │  - 人類行銷專員進行一鍵「確認/修改/駁回」           │
  └──────────────────────┬─────────────────────────┘
                         │ Approve
                         ▼
             [ 實際執行發信 API 對外傳送 ]
```

### 1. 第一關：鑑別式 AI 意圖與內容安全網閘 (Discriminative AI Guardrail)
在 LLM 輸出 `send_marketing_email` 的調用參數後，中台程式碼在呼叫真實 API 前，先啟動**鑑別式 AI 或是輕量級分類模型**進行語意安全過濾：
*   **幻覺防禦（事實核查）**：
    *   *機制*：使用鑑別式演算法，提取郵件正文中的商品名稱、折扣數字（例如「現折 500 元」）。
    *   *驗證*：與 CRM 行銷活動資料庫進行**硬規則比對**。如果 AI 憑空捏造了「送免費 iPhone」或未授權的「折價 50%」，鑑別層將直接攔截並報錯，要求 LLM 重新生成。
*   **敏感度與聲調審查 (Toxic Content Detection)**：
    *   *機制*：利用分類模型（如基於 BERT/DeBERTa 微調的安全分類器）審核 `email_body`。
    *   *目標*：檢查是否包含政治敏感、性暗示、年齡/性別歧視，或語氣過於激進等可能引起客戶反感或違反個資法的字眼。

### 2. 第二關：人工覆核機制 (Human-in-the-Loop)
**「這是企業資安防禦與品牌商譽的最關鍵底線。」**
*   **草稿與審批分開 (Draft-Review Separation)**：
    *   系統中 `send_marketing_email` 這個 API 函數的後端邏輯**並非直接發信**，而是將資料寫入資料庫的 `pending_marketing_emails`（待審郵件資料表）。
*   **行銷審查看板 (Ad-review Dashboard)**：
    *   在公司的 CRM 後台，為行銷人員提供一個「AI 行銷草稿審核看板」。
    *   人類行銷專員可以看見：
        *   *「客戶 ID / 真實姓名」*
        *   *「AI 推薦的理據（如：因為該客戶去年購買了長春青松防癌險）」*
        *   *「AI 生成的信件標題與內文」*
    *   **三種操作選項**：
        1. **一鍵批准**：確認無誤，系統正式呼叫 SMTP 發送郵件。
        2. **編輯修改**：人類專員微調錯字或潤飾語氣後發出。
        3. **駁回/重新生成**：告知 AI 駁回原因（如「優惠碼不適用此客戶」），讓 Agent 重寫。

## 📈 三、 商業決策價值總結

在 2026 年的企業級 Agent 應用中，「**Tool Calling 負責自動化效率，安全 Guardrail 負責法規風險，HITL 負責最終品牌聲譽**」。
透過將「自動執行」降級為「草稿推薦」，搭配地端鑑別式 AI 進行即時防禦，企業不僅能節省 80% 的行銷信件撰寫時間，更能 100% 避免因 AI 幻覺所導致的法律訴訟與公關災難，完美對齊 **ISO/IEC 42001（人工智慧管理系統）** 的主動風險控制規範。
