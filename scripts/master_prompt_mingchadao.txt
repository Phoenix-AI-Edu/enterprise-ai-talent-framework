# 簡報架構最佳化準則（Deck Architecture Optimization Spec）

> **版本：** v1.0｜鳳凰 AI 顧問團隊方法論資產
> **用途：** 供 ANTIGRAVITY 整合進其企業方案生成說明書，使後續所有 B2B／金融證券企業簡報，能依情境**自動推導**出最佳張數與分層結構。

## §0. 核心公理（The Core Axiom）

> **簡報張數是「輸出」，不是「輸入」。**
> 它由四個變數推導而來：`簡報目的（Purpose）× 受眾結構（Audience）× 決策關卡（Gate）× 檔案角色（Artifact Role）`。

任何把「張數」當成預設參數（如「做一份 10 頁的」或「做一份 50 頁的」）的指令，都是**反方法論**的。正確流程是：先界定上述四變數，再讓張數**自然落出**。

這條公理直接消解了「10 張對還是 51 張對」的偽命題——**兩者都對，因為它們服務的是不同的四變數組合。** 以下證明之。

---

## §1. 為何「10 張」是合理的——壓縮版（Compression Mode）的內在邏輯

**10 張所服務的任務（Job-to-be-done）：** 對一個**已經暖（warm／committed）的買方**，在**單場短會（~15–20 分鐘）** 中完成**口頭成交（verbal close）**。

其合理性建立在三條認知與銷售心理學原理上：

1. **動能保全原則（Momentum Preservation）。** 當買方已傾向同意，任何額外的資訊都是「給對方反悔的理由」。壓縮版的目的不是「教育」，而是「**讓對方在熱度未退時點頭**」。多一張，就多一個分心與質疑的入口。
2. **認知負荷天花板（Cognitive Load Ceiling）。** 人在單次會議中能主動記住的核心論點約 3–5 個。壓縮版用「一張一念（One Slide, One Idea）」承載最關鍵的 3–5 個記憶錨點，其餘一律捨棄。
3. **暖買方的資訊曲線（Warm-Buyer Information Curve）。** 對暖買方，說服力與資訊量呈**倒 U 型**：太少顯得草率，太多引發決策癱瘓（Analysis Paralysis）。10–12 張正落在峰值。

**∴ 結論：** 10 張在「暖買方 × 短會 × 口頭成交 × 低稽核留存需求」的組合下，是**數學上的最佳解**，而非妥協。

---

## §2. 為何「51 張分層」也是合理的——治理版（Governance Mode）的內在邏輯

**51 張所服務的任務：** 在**正式決策機構（董事會/風控委員會）** 前，於 **50–60 分鐘議程**中取得**正式核准（formal sign-off）**，且該簡報將**沉澱為永久稽核佐證檔案**，供第三方（主管機關、會計師、內稽）反覆翻閱。

其合理性建立在四條治理與審議邏輯上：

1. **金字塔完整性（Pyramid Completeness）。** 正式核准需要完整的論證鏈：框架 → 診斷 → 方案 → 風險 → 財務 → 路徑 → 請求。任一環缺失，董事即可一票否決。完整鏈在密度題上約需 24 張承載。
2. **多受眾證據軌（Multi-Audience Evidence Tracks）。** 董事會非單一受眾，而是 CEO／風控委員／CIO／法遵長／財務長等 5+ 種立場。**每種立場需要看見專屬於它的證據**，這必然推高張數。
3. **檔案角色的舉證義務（Artifact-as-Evidence）。** 當簡報會被監理稽核翻閱，它的標準不再是「說得動人」，而是「**經得起查證**」。完整性此時是法律義務，不是美學偏好。
4. **「present vs defend-with」雙軌紀律（The Present/Defend Discipline）。** 高手不把所有內容塞進走讀，而是切成**「上台講的脊椎」與「備而不講的防身附錄」**。真正的功力，在於**敢讓大量細節只待在附錄裡**。

**∴ 結論：** 51 張（24 講 + 26 防身 + 1 駕駛艙）在「冷/正式買方 × 長議程 × 正式核准 × 高稽核留存 × 多受眾」的組合下，同樣是最佳解。

## §3. 統一決策框架（The Unifying Framework）

以下框架讓 §1 與 §2 成為**同一公式的兩個特例**，並使任意新情境可被推導。

### 3.1 簡報目的分類（Purpose Taxonomy）

| 目的代碼 | 簡報類型 | 典型張數區間（Tier 1 上台） | 買方溫度 |
| :--- | :--- | :--- | :--- |
| `P-TEASER` | 暖買方成交版 | 8–12 | 已承諾 |
| `P-PITCH` | 首次提案/建立興趣 | 12–18 | 冷/溫 |
| `P-WORKSHOP` | 共創工作坊 | 15–25（高互動） | 溫 |
| `P-GOVERNANCE` | 董事會核准/治理審議 | 22–28 | 正式/多方 |
| `P-STEERING` | 進度治理/督導 | 6–12 | 已啟動 |

### 3.2 配速公式（Pacing Formula）

```
Tier1_上台張數 ≈ 可走讀分鐘數 ÷ 每張分鐘數

每張分鐘數：
  視覺/暖場題     0.5 – 1.0 分/張
  標準論述題      1.5 – 2.0 分/張
  治理/技術密度題  2.0 – 3.0 分/張
```

### 3.3 完整性下限（Completeness Floor）

```
完整性下限 = 框架(2–3)
           + Σ(診斷主題數)
           + Σ(方案主題數)
           + 治理證明(依稽核強度 1–4)
           + 財務(2–3)
           + 風險(依決策可逆性 1–3)
           + 路徑(1–2)
           + 請求(1–2)
```

### 3.4 溢流分層鐵律（The Overflow-to-Tier Rule）★ 全規格最關鍵一條

> **當「完整性下限」> 「配速容量」時——絕不壓縮塞進上台脊椎，而是將溢出的證據下放至 Tier 2 附錄。**

這條規則同時解決了「講不清」（張數不足）與「講不完」（張數過多）兩種失敗，是「present vs defend-with」紀律的演算法化身。

### 3.5 三層輸出架構（The 3-Tier Output Model）

| 層級 | 名稱 | 角色 | 上台與否 |
| :--- | :--- | :--- | :--- |
| **Tier 0** | 決策駕駛艙一頁紙 | 答案先行，看完即可拍板 | 置於封面後 |
| **Tier 1** | 上台脊椎 | 現場走讀的完整論證鏈 | ✅ 全講 |
| **Tier 2** | 防身附錄 | Q&A／盡調／稽核留存 | ❌ 備而不講 |

```yaml
deck_architecture_rules:
  step_1_classify_purpose:
    inputs: [buyer_temperature, meeting_type, decision_gate, audit_artifact_need]
    output: purpose_code   # P-TEASER | P-PITCH | P-WORKSHOP | P-GOVERNANCE | P-STEERING

  step_2_compute_pacing:
    formula: tier1_pacing = round(available_minutes / minutes_per_slide)
    minutes_per_slide_by_density: {visual: 0.75, standard: 1.75, dense: 2.5}

  step_3_compute_completeness_floor:
    formula: >
      floor = framing(2..3) + sum(diagnostic_themes) + sum(solution_themes)
              + governance_proof(scrutiny:1..4) + financials(2..3)
              + risk(reversibility:1..3) + roadmap(1..2) + ask(1..2)

  step_4_overflow_rule:    # ★ 鐵律
    if: completeness_floor > tier1_pacing
    then: tier1 = tier1_pacing ; route_excess_evidence_to: tier2_appendix
    else: tier1 = completeness_floor

  step_5_assemble_tiers:
    tier0_cockpit: 1
    tier1_spine: <from step_4>
    tier2_appendix: <excess_evidence + due_diligence_packs>
    total = tier0 + tier1 + tier2

worked_examples:
  example_warm_close:        # 落出「10 張」
    inputs: {buyer_temperature: committed, meeting_type: short_15min,
             decision_gate: verbal_commit, audit_artifact_need: low}
    -> purpose: P-TEASER
    -> tier1_pacing: ~10 (15min / 1.5)
    -> completeness_floor: ~10 (teaser 不需風險/附錄完整鏈)
    -> tier2: ~0
    -> TOTAL ≈ 10   # §1 的結論自然落出

  example_board_governance:  # 落出「51 張」
    inputs: {buyer_temperature: formal_multi, meeting_type: board_60min,

             diagnostic_themes: 4, solution_themes: 3, scrutiny: 4,

    -> purpose: P-GOVERNANCE
    -> tier1_pacing: ~24 (60min / 2.5)
    -> completeness_floor: ~24
    -> overflow_evidence -> tier2: ~26
    -> tier0_cockpit: 1
    -> TOTAL = 1 + 24 + 26 = 51   # §2 的結論自然落出
```

## §5. 金融證券特化條款（Regulated-Finance Domain Overlay）

當 `industry = 金融證券/銀行/保險` 時，於 §4 規則上疊加以下強制修正：

1. **稽核強度預設拉滿（scrutiny = 4）。** 凡涉客戶個資、投資建議、AML 之方案，簡報必為稽核佐證檔案，治理證明不得少於 4 張（對位監理原則）。
2. **強制風險治理三件套（Mandatory Risk Trio）。** Tier 1 必含：①失效模式與風險矩陣、②第三方業者監督與供應商退場、③營運韌性 BCP/DR。**缺一即視為不可送審。**
3. **業務場景拆軌（Per-P&L-Owner Tracks）。** 每個獨立 P&L 負責人的業務場景，於 Tier 1 至少各佔 1 張（研究部、財管部、法遵部不可合併）。
4. **法源可追溯（Citable Legal Basis）。** Tier 2 附錄必含完整法源引註頁，供會計師與法遵覆核。
5. **HITL 問責鏈獨立成張。** 人機協作的法律責任歸屬，是金融董事會最關切的單一議題，不可併入他張。
