## 【第一部分:前瞻戰略與業務診斷】

### 一、診斷總綱:三道牆其實是同一道牆——「沒有治理框架的 AI 投資」

三位列的三大硬傷——POC 死亡谷、ISO 稽核、自籌預算——表面是三個獨立問題,但作為顧問,我們必須指出它們的共同根源:

> **龍圖過去兩年燒掉近百萬做 AI,死在 POC,不是因為技術不夠好,是因為「先做技術、後想治理」的順序錯了。**

正確的智慧製造升級順序是:**先建治理框架(誰能用、用什麼資料、出錯誰負責、花多少錢觸發熔斷)→ 再讓技術長進框架裡。** 當您先有了 ISO 42001 的治理骨架,POC 自然不會死(因為現場師傅知道怎麼安全使用)、CIO 自然不防(因為 ERP 對接有唯讀緩衝與審計日誌)、CFO 自然放行(因為 API 有財務熔斷)、外銷稽核自然過關(因為治理文件本來就是 ISO 要的東西)。

**這就是本案的最高戰略:把 ISO 42001 治理框架當成「地基」,而不是「最後補的認證」。** 一塊地基,同時解決四個問題。

### 二、痛點一深度解讀:POC 死亡谷的三個真實死因

三位說「投入近百萬跟風做 AI,都死在 POC」。我們把這個死亡谷解剖成三個死因,因為對症才能下藥:

**死因一:現場師傅戴手套、滿手油污,無法打字報工報修。**
這是最常見也最致命的死因。任何需要一線師傅「停下手、脫手套、在平板上打字」的 AI 系統,在鍛造廠的油污高噪音環境裡都是死的。過去的 POC 把 AI 設計成「辦公室介面」,而鍛造現場根本不是辦公室。**解法不是訓練師傅打字,是讓 AI 用師傅本來就會的方式互動——講話。** 我們在第二部分會詳述邊緣端雙語 ASR 語音報工 + 複誦回路的設計。

**死因二:CIO 防範 AI 寫回鼎新 ERP,造成帳務與製程數據污染。**
這個防範**完全正確**,CIO 是對的。一個會直接寫入 ERP 的 AI,等於把您 17 年的製程數據與帳務暴露在「幻覺污染」風險下——AI 一旦寫錯一筆工單或庫存,污染會沿著 ERP 擴散到財報。**但 CIO 的正確防範被誤解成「不能用 AI」,這是過度防衛。** 正解是「非侵入式唯讀對接」:AI 只能讀 ERP、不能寫 ERP,所有 AI 產出的建議都先進「緩衝層」,經人工雙簽後才由人手動寫回 ERP。AI 看得到資料,但碰不到您的核心系統。

**死因三:CFO 防範 API 算力成本失控,拒批高昂帳單。**
這個防範**也完全正確**。雲端大模型 API 的計費是「用多少付多少」,沒有上限的設計會讓 CFO 在月底收到一張無法預測的帳單。一個工程師寫錯一個無限迴圈,可能一夜燒掉數十萬 Token。**正解是 API Gateway 財務熔斷**:設定單月預算上限、單次請求 Token 上限、迴圈偵測(Loop Guard),一旦逼近閾值自動降級或熔斷,並即時通知 CFO。把「無法預測的雲端帳單」變成「有天花板的固定成本」。

**戰略結論:** CIO 與 CFO 的防範不是阻礙,是龍圖的免疫系統。過去的 POC 失敗,是因為 SI 廠商把這兩位的正確防範當成「需要說服繞過的障礙」。鳳凰 AI 的做法相反——**我們把 CIO 與 CFO 的防線直接設計進架構裡**,讓他們從反對者變成這套系統的共同設計者。

### 三、痛點二深度解讀:ISO 42001 不是成本,是外銷合約的續命符

三位說歐洲大客戶要求年底前通過歐盟 AI Act 浮水印告知 + ISO/IEC 42001 認證,否則取消數千萬採購合約。

我們要把這件事的戰略意義講到位:**這不是一張「要花錢取得的證書」,這是「數千萬外銷合約的續命符」。** 算一筆帳——若因未通過認證失去這個歐洲客戶,單一年度損失數千萬;而建置 ISO 42001 治理框架 + DLP 遮罩 + 審計日誌的投資,相對於這個損失是零頭。這個 ROI 不需要滑桿模擬,它是「做 = 保住合約,不做 = 失去合約」的二元選擇。

更深一層的戰略洞察:**龍圖若能率先通過 ISO 42001,這會成為對抗岡山其他鍛造廠的差異化武器。** 當您的同業還在被歐洲客戶的稽核函嚇得手忙腳亂時,您已經能對客戶出具完整的 AI 治理報告——這讓龍圖從「供應商之一」升格為「客戶信任的策略夥伴」,在下一輪比價時擁有議價權。**合規,是龍頭企業甩開追兵的護城河。**

具體缺口,三位自己也點出了:「完全缺乏符合規範的機密數據遮罩(DLP)與審計日誌」。這兩項正是 ISO 42001 與歐盟 AI Act 的核心要求,也是第二部分的技術重點。

### 四、痛點三深度解讀:用政府的錢付轉型的帳

三位說全面引進智慧製造要數百萬,CFO 強烈反彈,除非能匹配政府補助並抵減營所稅。

這正是孟顧問的核心戰場。我們的策略是「**雙引擎降低自籌款**」:

**引擎一:SBIR / 產發署補助**——把龍圖的智慧製造升級包裝成「研發計畫」,對接經濟部 SBIR(小型企業創新研發計畫)或產發署智慧機械相關補助。以龍圖的營收規模與外銷地位,加上「ISO 42001 治理創新」這個切入點,過件競爭力強。但我們必須誠實:沒有任何顧問能保證 100% 過件,我們賣的是專業包裝與審查會潛規則對接,實質提高過件率,不是空頭保證。

**引擎二:產業創新條例第 10 條之 1 研發投資抵減**——這是更穩、更大的一塊。產創條例第 10 條之 1 允許企業將「智慧機械、第五代行動通訊、資通安全」相關投資,抵減當年度營所稅。龍圖的 AI 智慧製造升級若正確歸類,可申請投資抵減,直接降低營所稅負擔。**這部分不依賴審查會過件,是依法可享的租稅權益,確定性遠高於補助。** 關鍵在於支出的歸類與會計師簽證——這需要孟顧問與貴公司財務、簽證會計師三方協作,確保抵減項目經得起國稅局查核。

**戰略結論給 CFO:** 我們不是叫您「砸數百萬做 AI」,我們是規劃「**用 SBIR 補助 + 產創研抵,讓政府幫您付掉一大部分轉型款**」的財務閉環。CFO 的反彈是對的——而我們的工作就是把這個反彈變成「原來可以這樣付」的安心。

## 【第二部分:實務落地技術解決方案與安控】

### 一、整體架構:地端為主、混合雲為輔的「三閘三層」安控

龍圖的核心約束是「機密不出廠、ERP 不被寫、算力不失控」。我們設計的整體架構是「三閘三層」:

```
┌─────────────────────────────────────────────────────────┐
│  一線現場（鍛造/品管/包裝）                                  │
│  [語音報工] [光學檢測] [包裝視覺]  ← 邊緣裝置,完全地端       │
└───────────────────────┬─────────────────────────────────┘
                        │ 閘一:DLP 去識別化遮罩網閘
                        ▼
┌─────────────────────────────────────────────────────────┐
│  地端推論層（廠內 GPU 伺服器 / DMZ 隔離區）                  │
│  Taiwan-LLM / Llama 地端模型 + RAG 製程知識庫               │
└───────────────────────┬─────────────────────────────────┘
                        │ 閘二:ERP 唯讀緩衝層（只讀不寫）
                        ▼
┌─────────────────────────────────────────────────────────┐
│  鼎新 ERP（核心系統,AI 永遠碰不到寫入權限）                  │
└───────────────────────┬─────────────────────────────────┘
                        │ 閘三:API Gateway 財務熔斷（僅高階任務出雲）
                        ▼
┌─────────────────────────────────────────────────────────┐
│  雲端大模型（僅 CBAM 稅則解碼等高階任務,經 DLP 洗滌後出廠） │
└─────────────────────────────────────────────────────────┘
```

三閘:**DLP 遮罩閘**(機密出廠前去識別化)、**ERP 唯讀閘**(AI 只能讀不能寫)、**API 熔斷閘**(算力成本天花板)。三層:**地端優先**(機密任務全在廠內)、**混合雲**(僅高階任務經洗滌後出雲)、**人機雙簽**(關鍵決策人工覆核)。這個架構直接把 CIO 的 ERP 防線與 CFO 的成本防線**焊進系統**。

### 二、邊緣端雙語 ASR 語音報工 + 複誦回路(回應要求 2 前半)

#### 2.1 解放雙手:師傅用講的,不用打的

鍛造現場師傅戴手套、滿手油污、環境噪音大。設計核心是**讓師傅用講的完成報工報修**:

- **邊緣端 ASR(自動語音辨識)**:麥克風裝在工作站,師傅喊「3 號機,工單 A123,完成 500 件」,邊緣裝置在本地完成語音轉文字,**不上雲**(機密工單號不出廠)。
- **台國雙語對照**:支援台語、國語混雜口音與台語工業外來語(如特定機台術語),這對中高齡師傅至關重要。
- **VAD 靜音偵測 + 環境降噪**:先用 VAD 抓出「人在講話」的片段,再用降噪演算法剝掉鍛造機台的背景噪音,確保 85 分貝環境下辨識率達工業級。

#### 2.2 複誦回路:防止語音誤判造成報工錯誤

語音報工最大的風險是「聽錯」——師傅說 500,AI 聽成 5000。解法是**複誦回路(Read-back Loop)**:

```
師傅:「工單 A123,完成 500 件」
 AI :「複誦確認:工單 A123,完成 500 件,對嗎?」
師傅:「對」(或「錯,是 5000 件」)
 AI : 確認後才寫入緩衝層
```

這個複誦設計借鑑航管與醫療的標準作業——任何關鍵數字都要回讀確認。這也是 HITL(人在環)的最前線:師傅的「對」就是一次人工簽核。

### 三、DLP 去識別化遮罩網閘(回應要求 2 後半,ISO 42001 核心)

#### 3.1 機密出廠前的強制洗滌

任何資料在離開廠區地端、進入雲端大模型之前,必須通過 DLP 遮罩網閘。這道閘做的事,跟我們鳳凰 AI 內部的本地去識別化工具同一個原理:

- **結構化機密**:客戶採購單號、零件圖號、批號、報價、目的地車廠——用規則引擎(正則 + 格式庫)強制替換為佔位標籤(如 `CUSTOMER_1`、`PART_NO_1`)。
- **個資**:員工姓名、身分證、電話——同步遮罩,符合個資法第 8 條。
- **雙向遮罩**:出廠時遮罩、回廠時還原。雲端模型只看得到去識別化的結構,真實機密永不離開廠區。

#### 3.2 審計日誌:ISO 42001 的舉證命脈

ISO 42001 與歐盟 AI Act 的核心要求是「AI 的每個決策都要可追溯」。我們部署審計日誌:每一次 AI 推論記錄時間戳、輸入資料雜湊、模型版本、輸出結果、信心度、經手人員,保存供第三方稽核翻閱。**當歐洲客戶的稽核員到廠,您能在 10 分鐘內調出任一筆 AI 決策的完整軌跡——這就是通過 ISO 42001 的舉證能力。**

#### 3.3 歐盟 AI Act 浮水印告知

歐盟 AI Act 對「有限風險 AI」要求告知義務——AI 生成或輔助的內容須標示。我們在 AI 產出的報表、檢測報告上自動加註浮水印與告知聲明,確保符合 AI Act 透明度要求。

### 四、非侵入式 ERP 唯讀對接 + API Gateway 財務熔斷(回應要求 3)

#### 4.1 ERP 唯讀緩衝:讓 CIO 安心

- **唯讀對接**:AI 透過唯讀 API 或資料庫複本讀取鼎新 ERP,**永遠沒有寫入權限**。
- **緩衝層雙簽**:AI 產出的建議(如「建議補料」「異常工單」)先進緩衝層,經人工覆核雙簽後,由人手動寫回 ERP。AI 碰不到核心系統,污染風險歸零。
- **製程數據隔離**:AI 訓練用的製程數據存在獨立資料庫,與生產 ERP 物理隔離。

#### 4.2 API Gateway 財務熔斷:讓 CFO 安心

- **單月預算上限**:設定每月 API 花費天花板,逼近時告警、超過時熔斷。
- **單次 Token 上限**:每次請求設 Token 上限,防止單次異常請求暴衝。
- **Loop Guard 迴圈偵測**:偵測異常重複呼叫(如工程師寫錯的無限迴圈),自動阻斷。
- **智慧路由器**:簡單任務走地端免費模型,只有高階任務(如 CBAM 稅則解碼)才路由到雲端付費模型,把成本壓到最低。

這四道機制把「無法預測的雲端帳單」變成「有天花板的固定成本」,直接回應 CFO 的核心焦慮。

### 五、人機協作雙簽(HITL)總則

本案所有關鍵決策都採 HITL 分流:語音報工的複誦確認、ERP 寫回的人工雙簽、光學檢測紅燈的品管覆核、CBAM 申報的會計師簽證。**AI 永遠是建議者,人永遠是決策者。** 這不只是技術設計,更是 ISO 42001 要求的「問責鏈」——出錯時責任歸屬清楚,這是外銷稽核最關切的單一議題。
