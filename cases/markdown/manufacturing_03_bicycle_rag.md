# 某大廠 AI 落地戰略評估與「售後問答模組」RAG 工作流設計報告

## 📌 背景說明
本案委託單位為 某大廠（跨國自行車零件製造商）。目前面臨三大痛點：第一，售後服務手冊繁雜，經銷商每次提問技術細節，客服平均要花 30 分鐘查閱 PDF；第二，烤漆產線目前完全依賴人工肉眼檢測微小刮傷，員工工時長且容易漏檢；第三，採購合約散落各地，法務部門每年因為沒注意到過期的「自動續約與漲價條款」而損失數百萬元。

對照「十大企業級 AI 應用模組」，我們為其挑選最合適的 3 個 AI 應用模組進行導入評估，並將「RAG 企業知識庫問答系統」列為第一波 Quick-Win 試點專案。

## 🔄 「售後服務問答模組」RAG 核心工作流設計

為了確保 RAG 系統能精確檢索複雜的自行車零件規格（例如專有名詞與型號），我們為 某大廠設計了以下進階混合檢索與雙層分塊工作流：

### 1. 核心工作流 Mermaid 流程圖

```mermaid
graph TD
    %% 資料寫入端 (Data Ingestion)
    subgraph Data_Ingestion [資料寫入端 (Data Ingestion)]
        A[Bicycle PDF 手冊] --> B[1. 表格重構 Table Structuring]
        B --> C[2. 語意分塊 Semantic Chunking]
        C --> D{3. 雙層分塊 Parent-Child}
        D -->|大於 1000 Tokens| E[父分塊 Parent Chunks]
        D -->|100~200 Tokens| F[子分塊 Child Chunks]
        F --> G[4. 向量化 Vector Embedding]
        G --> H[(向量資料庫 Vector DB)]
        E -.->|映射關聯 ID| H
    end

    %% 使用者檢索端 (Retrieval & Generation)
    subgraph Retrieval_Generation [使用者檢索與生成端 (User Query)]
        I[經銷商/客服提問 <br> 'Bicycle-Hub-2026-X1 扭力規範'] --> J{5. Intent Router 意圖路由器}
        J -->|特定零件與型號| K[稀疏檢索 Sparse BM25]
        J -->|感性/口語化發問| L[稠密檢索 Dense Vector]
        
        K --> M[6. 混合檢索融合 Hybrid Fusion <br> alpha = 0.2]
        L --> M
        
        H -.->|檢索子分塊| M
        M --> N[7. 重排序 Re-ranking]
        N --> O[8. 父分塊上下文還原 Parent Retrieval]
        O --> P[9. 組合 Prompt 與安全護欄]
        P --> Q[10. 大模型推理 LLM Inference]
        Q --> R[無幻覺售後解答]
    end
```

### 2. 工作流步驟詳解
1. **表格重構 (Table Structuring)**：自行車手冊中含有大量的「螺絲扭力值、軸承尺寸」等表格。在寫入前，系統使用多模態 LLM 將 PDF 中的表格重構為 Markdown 格式，避免傳統 PDF 轉文字時的排版錯亂。
2. **語意分塊 (Semantic Chunking)**：以句為單位，計算相鄰句子的向量餘弦相似度。當相似度跌破動態閾值（如 0.82）時才切分，確保「步驟描述」不被粗暴地從中切斷。
3. **父子分塊 (Parent-Child Chunking)**：
   * 將手冊切分為小顆粒度的 **子分塊** (100~200 Tokens)，用於提高向量檢索的精準度。
   * 保留對應的 **父分塊** (1000~2000 Tokens，包含完整的上下文脈絡) 在資料庫中。
4. **混合檢索 (Hybrid Search) 與 $\alpha$ 動態調校**：
   * 由於經銷商的提問常包含精確的自行車零件型號（如 *Bicycle-Hub-2026-X1*），因此 **意圖路由器 (Intent Router)** 會在檢索時判定，並將 **$\alpha$ 權重下調至 0.2**。
   * 這代表系統會**極度偏重稀疏向量檢索 (BM25 關鍵字匹配)**，確保型號精確匹配，避免被 Embedding 向量空間的語意模糊化。
5. **父分塊還原**：當檢索命中子分塊後，系統會透過 ID 自動向上召回其對應的 **父分塊** 送入 LLM 上下文，提供完整的操作防護說明與警告資訊。

## 🛡️ 多維度防幻覺安全機制

為了確保 AI 客服不會對經銷商「胡謅規格」或「承諾非官方的保固條款」，我們設計了 **RAG 三元組評估框架 (RAG Triad)** 與防幻覺安全防線：

```text
               ┌──────────────────────────────────────────┐
               │              1. 使用者提問                │
               └──────────────┬────────────────────▲──────┘
                              │                    │
                Context       │                    │ Answer
              Relevance       ▼                    │ Relevance
               ┌──────────────┴───────────┐        │
               │   2. 檢索到的上下文資訊   ├────────┴──────┐
               └──────────────┬───────────┘                │
                              │                    ┌───────┴──────┐
                              │ Groundedness       │ 3. 模型回答   │
                              └───────────────────►└──────────────┘
```

### 1. RAG 三元組評估防線 (RAG Triad)
在 LLM 生成回答前，系統會在後台啟動三個獨立的微型鑑別器進行評估，不合格者將直接攔截：
* **上下文相關性評估 (Context Relevance)**：檢查檢索出的 PDF 手冊分塊，是否真的包含與提問（例如「扭力規格」）相關的內容。若無，代表知識庫無此資料，直接輸出標準回覆：「抱歉，官方手冊無此零件扭力資料，已為您派接真人客服。」
* **真實性評估 (Groundedness / Hallucination Grader)**：檢查模型生成的回答，其中每一句話是否都能在檢索到的 PDF 上下文中找到**直接的文字支持**（檢查是否有模型自我發揮的幻覺）。
* **回答相關性評估 (Answer Relevance)**：檢查最終生成的回答是否切中經銷商的提問，避免模型答非所問。

### 2. 提示詞工程約束 (Prompt Grounding & Negative Constraint)
在 System Instruction 中加入最嚴格的**負向約束條件**：
```text
【系統角色與原則】
你是一位專業的自行車售後技術支持助理。
1. 你「只能」依據下方 [Provided Context] 內的內容進行回答。
2. 如果 [Provided Context] 中的資訊不足以回答使用者的提問，你必須說：「抱歉，目前提供的手冊中沒有相關的技術參數，我已為您記錄並轉接技術支援人員。」，絕對不允許進行任何推論或猜測。
3. 嚴禁承諾任何退款、折扣、延長保固或退換貨條件。
```

### 3. 可追溯的審計日誌與引用機制 (Citation & Lineage)
* **自動標註引用源**：模型輸出之所有規格數據，必須強制以 `[手冊名稱][頁碼][段落]` 的格式進行標註（如：*建議鎖緊扭力為 6 Nm [經銷商手冊-2026版][Page 14]*）。
* **安全網閘攔截**：如果模型輸出的內容中含有數據（如數字、溫度、扭力），但該輸出沒有成功關聯到上下文的引用，系統後台會直接熔斷該次輸出，防止幻覺數據外洩。

### 4. 人機協作 (HITL) 與 OOD 熔斷機制
* 當 RAG 三元組的綜合置信度分數（Confidence Score）低於 **0.78** 時，系統將自動啟動 **HITL (Human-in-the-loop) 流程**。
* 此時系統不會將 AI 的回答直接發送給經銷商，而是將「提問 + AI 客稿 + 檢索到的 PDF 頁籤」推送至後台的真人客服審核介面（Validator 模式），由客服人員確認或修改後再行送出。
